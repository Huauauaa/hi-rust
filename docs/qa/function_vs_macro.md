# 函数与宏的区别

初学 Rust 时，很容易把 `println!` 当成普通函数。带 `!` 的是**宏**（macro），用 `fn` 定义的是**函数**（function）。两者都能复用代码，但工作方式不同。

## 一句话区分

| | 函数 | 宏 |
|---|------|-----|
| 定义 | `fn name(...) { ... }` | `macro_rules! name { ... }` 等 |
| 调用 | `name(args)` | `name!(args)` |
| 执行时机 | 运行时调用 | **编译期展开**成普通 Rust 代码 |
| 类型检查 | 对函数签名检查 | 对展开后的代码检查 |

## 函数：运行时的常规逻辑

函数在编译后成为可调用单元，参数和返回值类型在签名里写死：

```rust
fn add(a: i32, b: i32) -> i32 {
    a + b
}

let sum = add(1, 2); // 运行时执行 add
```

- 参数个数、类型必须匹配。
- 函数体就是最终执行的代码，不会被「替换」成别的形状。

详见语法章节：[函数](/syntax/functions)。

## 宏：编译期的代码生成

宏在**编译前**被展开。你写的 `macro!` 调用会变成一段新的 Rust 代码，再交给编译器做类型检查。

```rust
println!("x = {x}");   // 宏调用
vec![1, 2, 3];         // 宏调用
```

以 `vec!` 为例，编译器大致把它展开成类似下面的代码（简化示意）：

```rust
// vec![1, 2, 3] 展开后近似为：
{
    let mut temp = Vec::new();
    temp.push(1);
    temp.push(2);
    temp.push(3);
    temp
}
```

宏可以根据**传入的 token** 生成不同代码，这是函数做不到的——函数只能接收已确定类型的值，不能根据「写法」改变结构。

## 常见对比

### 可变参数

函数参数个数固定；宏可以按调用处写法接受不同数量参数：

```rust
println!("one");                    // 一个参数
println!("{} + {} = {}", 1, 2, 3);  // 多个参数
```

若用函数实现同样灵活性，通常要借助 trait 或多次重载，写法更重。

### 编译期分支

宏可以在展开时根据传入内容生成不同代码，例如 `format!` 根据格式串生成对应的格式化逻辑。函数只能在运行时根据值做分支。

### 类型与错误信息

函数的类型错误直接指向函数签名和调用处。宏先展开再检查，报错有时出现在「展开后的代码」里，栈信息可能更绕，这是宏的常见缺点。

## 为什么标准库大量用宏？

1. **语法糖**：`vec!`、`format!`、`assert!` 让常用模式写起来更短。
2. **零成本抽象**：展开后往往是内联的普通代码，不必然增加运行时开销。
3. **无法用函数表达的语法**：如 `?` 配合的错误传播、部分 DSL 式写法，依赖宏在编译期改写代码。

日常写业务逻辑，**优先用函数**；需要重复代码模式、可变参数或与语法深度结合时，再考虑宏（多数情况直接用标准库提供的宏即可）。

## 宏的种类（了解即可）

| 种类 | 说明 |
|------|------|
| **声明式宏** `macro_rules!` | 模式匹配式替换，如 `vec!`、`println!` |
| **过程式宏** | 在 AST 上操作，如 `#[derive(Debug)]`、属性宏、函数式宏 |

过程式宏多用于派生 trait、编译期代码生成，入门阶段知道 `derive` 是宏即可。

## 小结

| 场景 | 更合适的选择 |
|------|----------------|
| 普通业务逻辑、可测试的单元 | 函数 |
| 固定 API、清晰类型 | 函数 |
| 可变参数、格式串、断言等语法糖 | 标准库宏 |
| 为类型自动实现 trait（如 `Debug`） | `derive` 等过程式宏 |

记住调用时的 `!`：有叹号是宏，在编译期展开；无叹号且用 `fn` 定义的是函数，在运行时执行。
